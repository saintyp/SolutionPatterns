Cloud Design Principles

The cloud design principles outline the characteristics of an application that we want to follow. They are based on both the lifespar and 12 factor models but are tailored to our organisational needs.

Use the principles when designing your application to produce a solution that is tailored towards cloud native practices.

Principle Explanation Theme
Cloud Product first Think about using cloud native tools and cloud native architecture first Strategic
Keep it simple Don't over complicate it unless you need to Strategic
Configure everything as code Don't do any manual changes right code for everything Design
Design for failure Expect elements of you service to fail. Use patterns that include circuit breakers Design
Design for disposable/Stateless resources Don't store anything on the instance or in the service which would be lost if te instance failed. Design
Design for loosly coupled behaviour Design your application so that it is not a monolith. Don't have components that rely on each other Design
Design for security Conform to security policies. Apply security at all layers in the design Design
Design for scalability Scale horizontally to increase availablity Design
Design for data usage requirements Understand the needs of your data and whether it needs to be persistent, transient, or for consumption in data analytics. Design
Everything consumed as a service All things that you need to interact should be treated as a service i.e logging as a service where you you have a set of defined. With a contract and sla Design
Externalise all configuration and data Don't tie config and data to where the application code runs Design
Design availability into the applications Don't rely on infrastructure to provide availablity Design
Tag/Label every resource Tag and label every resource so you can track its usage. Instrument everything. Everything should be easily identifiable. Make tags relevant to billing and audit Design
Optimise for performance & cost Optimise the solution to the best performance for the least cost Design
Test continuously From development through to production you should be constantly testing for resilience and performance(for performance try to test at scale). Test your automated recovery processes. Design
Enable traceability: Log everything Log everything that is happening in the environment Design
Make every process run in its own instance Don't have an instance perform two functions separate them out Design
