Data

More than likely your application will require to retrieve or store some data. You need to consider how this is done is a cloud friendly way as this can be one of the major blockers for high availability and scalability within applications. The tools and techniques that you could employ vary depend of what you are trying to achieve

Persisting data

If you need to store data then you have the broad choice of two styles either relational or non-relational style databases.

Relational database

A relational database (the concept) is a data structure that allows you to link information from different 'tables', or different types of data buckets. A data bucket must contain what is called a key or index (that allows to uniquely identify any atomic chunk of data within the bucket). Other data buckets may refer to that key so as to create a link between their data atoms and the atom pointed to by the key.

Non-relational database

A non-relational database just stores data without explicit and structured mechanisms to link data from different buckets to one another. Examples of relational databases, sql server, MySQL, Amazon RDS Examples of non-relational databases, mongo dB, document dB,

Transient data

This maybe data that you need to hold on a customer as part of their journey or maybe data that you need to enrich the application. If the data was lost it could be repopulated easily and would not affect the fundamentals for the application

Transient data can be stored in caches (like memcache, redis, appfabric) for fast retrieval or in message queues for further processing.

Analytics

This is primary data that will be used for data mining to produce MI or customer insights. This data just needs to stored in the correct format but not necessarily in a database. Most pass systems can happily read this data from an object or block storage.

Backup/Restore

It is important in your solution design to include provision for what happens if it goes wrong. For example, is this the primary source of the data or could it be reconstituted from another primary source (and how long would this take). What would be the effect in application downtime of data corruption? Do you need to keep secondary copies of your data in case of corruption.

What does good like for data?

The choices you make in the section are very personal to your application. However the core factors you need to consider are • What type of data am I storing (transactional, key-value, data for analytics) • Will my solution scale • What would happen if my data were corrupted/manipulated • How fast do you need to retrieve the data • Do I need to persist the data or use it transiently
