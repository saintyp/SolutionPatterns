Application Lifecyle Management (ALM)

Continous delivery/continuous integration

To work effectively in the cloud your whole system needs to be able to be delivered and updated quickly. The most widely adopted method of this is to setup a delivery pipeline (using something like Jenkins). This will automate not only the delivery of the code but also the build and maintenance of the infrastructure. If can also be used to provide elements of the compliance/security required in the cloud.

If the pipeline is the control point and nothing else can deploy to your cloud infrastructure then you can move a lot of compliance and security to this pipeline thus simplifying the ongoing maintenance of the platform.

Automation Testing

With automated release comes the need for automated testing. It is important that the application is tested at multiple points throughout its journey in the pipeline. This is to ensure that the application released into production functions correctly

Automatic release management

To do continuous delivery it is important that code/infra can get released to production with minimal human intervention. This element is about not only about automating code release through the pipeline but also automating approvals via change management.

Roll back. Your application needs to function so that any release to production can be rolled back to a previous version without the need to data purge. This is normally achieved by decoupling the database upgrades from the application upgrades so that each can be released asynchronously.

ALM what might good look like?

• Jenkins pipeline • Infrastructure and environment defined in code. • Calls from the pipeline to automate testing of the application for unit testing and integration testing. • Calls from pipeline to verify base image compliance • Calls from pipeline for change and release management • Calls to deployment mechanism (i.e. ansible) Code designed in a way that it can be released separately and rolled back without issue.
